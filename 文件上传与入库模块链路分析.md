# Auto OnCall RAG 文件入库链路详解

# 1\. 整体架构

当前项目已经具备一个完整的：

```text
文件上传型 RAG 知识库系统
```

核心流程如下：

```text
用户上传文件
    ↓
file.py（文件上传模块）
    ↓
vector\_index\_service.py（索引控制层）
    ↓
document\_splitter\_service.py（文档切分层）
    ↓
vector\_store\_manager.py（向量存储层）
    ↓
Milvus（向量数据库）
```

最终实现：

```text
文件 → 文本切分 → embedding → 向量存储 → RAG 检索
```

\---

# 2\. file.py 文件上传模块

## 模块作用

这是整个系统的上传入口。

主要负责：

* 接收用户上传文件
* 校验文件是否合法
* 保存文件到 uploads 目录
* 自动触发向量索引
* 返回上传结果

\---

## 上传流程

整体流程：

```text
用户上传文件
    ↓
校验文件名
    ↓
校验扩展名
    ↓
检查文件大小
    ↓
保存到 uploads
    ↓
自动触发向量索引
```

\---

## 文件类型限制

当前系统只允许：

```text
.txt
.md
```

原因是目前系统主要处理纯文本内容。

这样可以降低：

* 文本解析复杂度
* embedding 异常
* 文件格式兼容问题

\---

## 文件大小限制

当前限制为：

```text
10MB
```

主要作用：

* 防止上传超大文件
* 避免 embedding 时间过长
* 降低内存占用
* 防止接口超时

\---

## 文件名清洗

系统会自动处理：

* 空格
* 非法字符
* Windows/Linux 路径问题

例如：

```text
my file?.txt
```

会被处理为：

```text
my\_file\_.txt
```

这样可以保证文件系统安全。

\---

## 自动创建知识库索引

上传成功后，系统会自动调用：

```text
vector\_index\_service
```

继续执行：

```text
文件读取
↓
文本切分
↓
embedding
↓
Milvus 入库
```

也就是说：

```text
上传文件 = 自动进入知识库
```

\---

# 3\. vector\_index\_service.py 索引控制层

## 模块作用

这个模块是整个 RAG 入库流程的：

```text
总控制层
```

它负责：

* 读取文件
* 调用文本切分
* 调用向量存储
* 管理索引流程
* 处理批量索引

它本身不负责：

* embedding
* Milvus 操作
* 文本切分算法

而是负责协调各个模块。

\---

## index\_single\_file()

这是单文件索引核心方法。

执行流程：

```text
读取文件
    ↓
转换为 Document
    ↓
调用 document\_splitter\_service
    ↓
生成 chunk
    ↓
调用 vector\_store\_manager
    ↓
写入 Milvus
```

\---

## 为什么需要这一层？

如果没有这一层：

```text
上传模块
直接操作 embedding
直接操作 Milvus
```

会导致：

* 模块耦合严重
* 不易维护
* 不易扩展

因此需要：

```text
索引控制层
```

统一管理整个入库流程。

\---

## index\_directory()

这个方法用于：

```text
批量索引整个目录
```

适合：

* 初始化知识库
* 重建索引
* 批量导入历史文件

流程：

```text
遍历目录
    ↓
逐个读取文件
    ↓
逐个建立索引
```

\---

# 4\. document\_splitter\_service.py 文档切分层

## 模块作用

这个模块负责：

```text
把长文本切分为多个 chunk
```

这是 RAG 系统中非常重要的一步。

\---

# 为什么需要切分？

LLM 和 embedding 模型都不适合直接处理超长文本。

例如：

```text
一个 50 页文档
```

如果直接 embedding：

* 成本高
* 检索精度低
* 语义不聚焦

因此需要：

```text
大文本 → 小 chunk
```

\---

## 切分后的效果

例如原文：

```text
公司报销制度全文
```

切分后：

```text
chunk1：报销流程
chunk2：审批规则
chunk3：发票要求
chunk4：到账时间
```

这样在用户提问：

```text
“报销多久到账？”
```

时，只会检索：

```text
到账时间相关 chunk
```

提高检索精度。

\---

## chunk 的核心作用

chunk 是 RAG 检索的最小单位。

后续进入 Milvus 的并不是整个文件，而是：

```text
chunk
```

\---

## metadata

切分时还会给每个 chunk 添加 metadata。

例如：

```text
来源文件
chunk 序号
文件名
```

后续：

```text
删除旧数据
来源追踪
```

都会依赖 metadata。

\---

# 5\. vector\_store\_manager.py 向量存储层

## 模块作用

这是整个系统的：

```text
向量数据库操作层
```

负责：

* 初始化 Milvus
* embedding + 向量写入
* 删除旧数据
* 相似度检索

\---

## 模块位置

它位于：

```text
LangChain ↔ Milvus
```

之间。

属于：

```text
数据访问层（DAL）
```

\---

## 初始化 Milvus VectorStore

系统启动时会创建：

```text
LangChain Milvus VectorStore
```

主要配置包括：

* collection\_name
* embedding\_function
* vector\_field
* text\_field
* metadata\_field

\---

## embedding\_function

这是整个系统最关键的配置之一。

作用：

```text
文本 → 向量
```

例如：

```text
“报销多久到账”
↓
\[0.12, -0.55, ...]
```

生成的向量最终存入 Milvus。

\---

## add\_documents()

这个方法负责：

```text
批量向量化并写入 Milvus
```

内部流程：

```text
Document
↓
提取文本
↓
调用 embedding
↓
生成向量
↓
写入 Milvus
```

这是整个 RAG 入库最核心的一步。

\---

## 为什么使用 LangChain Milvus？

因为它帮你封装了：

* embedding 调用
* 向量写入
* similarity\_search
* collection 管理

否则需要直接操作 PyMilvus，代码复杂度会高很多。

\---

## delete\_by\_source()

这个方法负责：

```text
删除某个文件对应的所有向量数据
```

主要用于：

```text
文件重新上传
→ 覆盖旧索引
```

删除依据：

```text
metadata.\_source
```

因此 metadata 非常重要。

\---

## similarity\_search()

这是整个 RAG 检索核心。

执行流程：

```text
用户问题
↓
embedding
↓
query vector
↓
Milvus ANN 搜索
↓
返回最相似 chunk
```

例如：

```text
“报销多久到账？”
```

会检索出：

```text
“财务将在三个工作日内处理”
```

相关 chunk。

\---

# 6\. Milvus 向量数据库

## 模块作用

Milvus 是整个系统的：

```text
向量数据库
```

负责存储：

* embedding 向量
* 文本 chunk
* metadata

\---

## 为什么使用向量数据库？

传统数据库：

```text
只能做关键词匹配
```

而向量数据库支持：

```text
语义相似度搜索
```

例如：

```text
“报销多久到账”
```

即使知识库写的是：

```text
“财务会在三个工作日完成打款”
```

也能匹配成功。

\---

## ANN 搜索

Milvus 内部使用：

```text
ANN（Approximate Nearest Neighbor）
```

进行高性能向量检索。

适合：

```text
百万级向量搜索
```

# 8\. RAG 检索链路

用户提问时：

```text
用户问题
    ↓
query embedding
    ↓
Milvus similarity\_search
    ↓
返回相关 chunk

LLM 生成答案
```

这就是完整的：

```text
RAG（Retrieval-Augmented Generation）
```

流程。
